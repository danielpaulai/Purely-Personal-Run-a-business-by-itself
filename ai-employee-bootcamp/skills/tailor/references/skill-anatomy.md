# Skill Anatomy
# Tailor Reference, Purely Personal Bootcamp

Every skill the Tailor produces must follow this exact 8-section structure.
Do not skip sections. Do not reorder them. Do not truncate.

---

## THE 8-SECTION SKILL STRUCTURE

1. YAML Frontmatter
2. Skill Header + WHO Block
3. Reference File List
4. Voice Lock (replaces Voice DNA Check)
5. Intake Form
6. Step-by-Step Workflow
7. Delivery Format
8. Quality Gate

---

## SECTION 1, YAML FRONTMATTER

```yaml
---
name: [skill-name-in-kebab-case]
description: >
  [What the skill does + who it's for + when to trigger it.
   Must name: the participant, their ICP role/industry, their offer type.
   Must include all trigger phrases. Target: 80-120 words.]
---
```

Every description must include:
- 3-5 exact trigger phrases ("write a post", "DM sequence", etc.)
- The participant's name or brand
- The ICP so the skill is clearly not for everyone
- The output so the user knows what they're getting

---

## SECTION 2, SKILL HEADER + WHO BLOCK

```markdown
# [Skill Name], [PARTICIPANT NAME]
# Fitted by Purely Personal · by Daniel Paul

WHO THIS IS FOR:
[PARTICIPANT NAME] is a [their role/title] who serves [ICP description].
Their offer: [OFFER NAME], [one-sentence transformation delivered].
Their unique angle: [UNIQUE ANGLE, what they say that no competitor says].
Their ICP: [role] at [company stage] dealing with [top pain in ICP's own words].
Competitors to avoid sounding like: [Top 3 by name].

One job: [skill's mission statement, specific to this participant].
If the output could have been produced by any [niche] [skill type], rebuild it.
```

## SECTION 3, REFERENCE FILE LIST

```markdown
Read ALL reference files before writing:
- /references/voice-dna.md, [PARTICIPANT]'s voice rules, ICP, banned words, hook archetypes
- /references/human-writing-standards.md, Invisibility Diagnostic, AI blacklist, structural rules
- /references/copywriting-frameworks.md, 8 frameworks, when to use each, examples
- /references/design-system.md, brand tokens for formatted output
- /references/scrub.md, 9-scrub system, run on every output before delivery
- /references/[participant]-context.md, ICP, offer, proof points, competitors
```

---

## SECTION 4, VOICE LOCK

Replaces the generic "Voice DNA Check." Participant's voice is the default. No fallback.

```markdown
## VOICE LOCK

This skill writes in [PARTICIPANT]'s voice. Always. No fallback. No override.

Voice rules (non-negotiable):
1. [Rule 1, specific, not generic]
2. [Rule 2]
3. [Rule 3]
4. [Rule 4]
5. [Rule 5]

Banned words (remove from every output before delivery):
[Exact banned list, minimum 10 words/phrases]

Sentence rhythm: [description]
Energy level: [1-5 with their level]
CTA format: [their exact format]

If any output violates these rules, rewrite before delivering.
```

---

## SECTION 5, INTAKE FORM

Pre-fill everything from the Tailor Briefing. Only ask for session-specific unknowns.

**Pre-fill:** participant name/role, ICP, offer name/transformation, proof points, competitors, voice rules
**Ask per session:** the specific topic/idea/prospect/use case for this session only

---

## SECTION 6, STEP-BY-STEP WORKFLOW

Keep the same steps as the starter skill. Replace all generic placeholders with specific context.
- "your ICP" → "[PARTICIPANT]'s ICP: [specific description]"
- "your offer" → "[OFFER NAME]"
- "your voice" → "[PARTICIPANT]'s voice rules from Section 4"
- Research steps must name specific niche and competitors

Never remove or reorder steps. Add context into each step. Never add new steps that break the flow.

---

Update the header to show participant's name and brand:

[PARTICIPANT NAME], [OUTPUT TYPE]
[Participant's brand or company name]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Metadata line must reflect participant's standards:
```
Framework: [name] · Hook: [archetype] · [char count] chars
Score: [N]/4, Voice [✓/✗] POV [✓/✗] Specific [✓/✗] Asking [✓/✗]
Voice check: [PARTICIPANT]'s rules applied ✓
Banned words: none detected ✓
Scrub: all 9 applied ✓
```

---

## SECTION 8, QUALITY GATE

Keep all original skill checks. Append participant-specific checks AND all 9 scrub checks:

```
PARTICIPANT-SPECIFIC CHECKS:
□ Voice: matches [PARTICIPANT]'s Voice DNA
□ ICP: written specifically for [ICP ROLE] at [COMPANY STAGE]
□ Language: uses ICP's own words
□ Offer: output connects to [OFFER NAME]
□ Sounds nothing like: [COMPETITOR 1], [COMPETITOR 2], [COMPETITOR 3]
□ Proof: [PARTICIPANT]'s proof points woven in where relevant
□ Banned: [TOP 3 BANNED WORDS], confirmed absent
□ CTA: follows [PARTICIPANT]'s format

SCRUB CHECKS:
□ Scrub 1: zero em dashes (, ), scan full output
□ Scrub 2: zero banned words, checked against references/scrub.md
□ Scrub 3: zero banned openers, transitions, meta-commentary
□ Scrub 4: zero banned closers, sentence enders, dramatic pauses
□ Scrub 5: no banned rhetorical devices or contrast patterns
□ Scrub 6: no AI structural patterns, three-part, Hook-Bridge-Value-CTA broken
□ Scrub 7: specificity passed, no vague scale language, no uncited stats
□ Scrub 8: tonal patterns checked, no relentless positivity, no conflict avoidance
□ Scrub 9: no formatting tells, emoji, hashtags, line breaks feel human
□ Final test: would [PARTICIPANT] read this and say "I wrote that"?
```

---

## MINIMUM VIABLE FITTED SKILL CHECKLIST

□ YAML description names participant, ICP, and offer
□ WHO block is filled with specific context, no generic placeholders
□ Reference files include scrub.md
□ Voice Lock has specific rules from Voice DNA, not "write conversationally"
□ Intake form pre-filled, only asks for session-specific unknowns
□ Every workflow step has specific context replacing generic placeholders
□ Delivery format header shows participant's name and brand
□ Quality gate includes participant-specific checks AND all 9 scrub checks
□ No instance of "your ICP", "your offer", "your audience" remains
□ The skill could not be installed by a different participant without editing
