# PARA · The Second Brain Knowledge Architecture

> Most operators have 1,000+ files scattered across 12 tools. Can't
> find anything. Re-create work that already exists. PARA is the
> 4-folder system that ends that. 4 folders. Top level. Everywhere.
> Forever.

---

## The 4 folders (that's it · only 4)

```
P · PROJECTS    · active · with deadlines · tied to current Rocks
A · AREAS       · ongoing · standards to maintain (not finishable)
R · RESOURCES   · topical reference · saved for future use
A · ARCHIVE     · completed Projects · inactive Areas · dormant Resources
```

Apply the 4 folders TOP-LEVEL in:
- Notion (your CRM, content calendar, business docs)
- Drive (your files, newsletter drafts, weekly wraps)
- Local computer (Documents folder)
- Even Gmail (labels)

Same 4 folders. Same names. Same logic. Everywhere.

---

## P · PROJECTS · active work with deadlines

A Project is a **specific outcome with a deadline** that you're
actively working toward.

**Examples:**
- Workshop · May 2-3 launch
- AI CEO cohort · 30 sign-ups
- Q2 newsletter relaunch
- Client engagement · Acme Corp · Q2 retainer
- Frameworks library buildout

**Rules:**
- Has a clear DONE state
- Has a deadline (or rough timeline)
- Currently has your attention this week or month

**Naming convention:** `[outcome] · [deadline]`

- "30 cohort members · May 3"
- "Frameworks library shipped · April 30"
- "Newsletter relaunch · Q2 2026"

When a Project completes (the outcome happens) · MOVE it to ARCHIVE.

---

## A · AREAS · ongoing standards (not finishable)

An Area is a **standard you maintain over time.** Not a project. Not
finishable. Just sustained.

**Examples:**
- Marketing · ongoing content + audience growth
- Sales · ongoing pipeline + revenue
- Operations · ongoing systems + SOPs
- Cash · ongoing financial health
- Leadership · ongoing decision quality
- Health · ongoing fitness + sleep + energy
- Family · ongoing relationships
- Network · ongoing professional connections

**Rules:**
- NOT time-bound (no deadline)
- Has a quality standard you're maintaining
- You'd be unhappy if it deteriorated

**The 5 engines map directly to 5 Areas.** Each engine maintains its
respective Area.

---

## R · RESOURCES · topical reference for future use

A Resource is **knowledge you might use someday** but isn't tied to
a Project or Area right now.

**Examples:**
- "Hook patterns for LinkedIn"
- "AI tools roundup"
- "Workshop facilitation methods"
- "Pricing case studies"
- "Books to read"

**Rules:**
- Topical (organized by SUBJECT, not by Project)
- Saved for future · not currently needed
- You'd want to find it again later

When a Resource starts being USED (because you started a Project
that needs it) · pull a copy into the Project. Don't move the
Resource itself.

---

## A · ARCHIVE · everything no longer active

The Archive is for:
- Completed Projects (the workshop is over · move it)
- Inactive Areas (you stopped doing podcasting · move it)
- Resources you no longer need

**Rules:**
- Don't delete · move to Archive
- Searchable later if needed
- Frees mental load (it's "out of sight")

**Why not delete?** Because in 18 months you'll need that one note
from that one Project. Archive · don't delete.

## How to file ANYTHING in 5 seconds

The PARA filing question:

```
Is this for a current PROJECT?         → P
Is this for an ongoing AREA?            → A (Areas)
Is this topical reference for future?   → R
Is this no longer active?               → A (Archive)
```

That's it. Most knowledge workers spend 15-30 minutes deciding where
to save things. PARA reduces that to 5 seconds.

## The actionability rule (the secret sauce)

PARA's superpower: **organize by ACTIONABILITY, not by topic.**

**Wrong (topical):**
```
Marketing/
  LinkedIn ideas.md
  Newsletter drafts/
  Hook patterns.md
  Workshop launch.md
  Workshop launch checklist.md
```

You can't tell what's active vs reference vs done.

**Right (PARA):**
```
P · Projects/
   Workshop launch · May 3/
     ↳ launch checklist.md
     ↳ actively-drafted-newsletters/

A · Areas/
   Marketing/
     ↳ LinkedIn ideas (ongoing)

R · Resources/
   Hook patterns/
     ↳ saved templates

A · Archive/
   Q1-2026 newsletter relaunch/   (completed)
```

Now you can see at a glance: what's active, what's ongoing, what's
reference, what's done.

## Migration · how to move FROM chaos TO PARA

Don't try to migrate everything at once. Use the **just-in-time
rule:**

1. **Set up the 4 folders top-level** in your main tool (Notion or
   Drive)
2. **Going forward**, file new things into PARA
3. **As you need to USE old files**, move them to PARA on the way to
   using them
4. **Don't worry about the 1,000 old files.** They'll migrate
   themselves over 6 months.

After 6 months · 80% of what you actually USE will be in PARA. The
rest stays in legacy folders, slowly forgotten · which is correct.

## Project · Area mapping in PARA

Every Project should belong to ONE Area.

```
PROJECT: "30 cohort members · May 3"
  ↓ part of
AREA: "Sales · ongoing pipeline + revenue"

PROJECT: "Frameworks library shipped"
  ↓ part of
AREA: "Operations · ongoing systems + SOPs"
```

When a Project completes, the Area lives on. The Area is the
sustained standard. Projects are bursts of work toward that standard.

## Weekly review · the PARA maintenance ritual

Once a week (Friday is best):

```
1 · PROJECTS  · which Projects are active right now?
                · which can I close (move to Archive)?
                · which need new attention?

2 · AREAS     · which Areas am I maintaining well?
                · which are slipping?
                · do I need a Project to fix a slipping Area?

3 · RESOURCES · anything to file from this week?
                · anything to clear out?

4 · ARCHIVE   · anything to move into Archive?
```

15-30 minutes. Compounds. After 12 weeks, your Second Brain runs
itself.

## Apply it to /operations-engine

When the engine generates the Friday Wrap:

1. **Audit Projects** · which closed this week? which advanced? which
   stalled?
2. **Audit Areas** · which are healthy? which are slipping?
3. **Surface inactive Projects** that should move to Archive
4. **Surface stale Resources** that haven't been touched in 6+ months
5. **Generate the next-week Project list** based on Areas that need
   attention

When the engine generates SOPs (operations-engine output):

1. SOPs are filed in **Resources** by topic
2. SOPs that get used regularly get linked from **Areas**
3. SOPs tied to current work get pulled into **Projects**

When the engine creates the BUSINESS-BRAIN.md (build-my-brain):

1. The 9 sections of BRAIN map to 9 sub-Areas under "Business" Area
2. Updates to BRAIN trigger updates to relevant Projects
3. Frameworks library lives under Resources
4. Completed cohorts move to Archive

## Why PARA beats other systems

**Folders by year (2024 / 2025 / 2026):** can't find current work
because it's mixed with last year's done work.

**Folders by client:** breaks when you switch clients · and doesn't
work for ongoing Areas.

**Tag-only systems:** infinite tagging is decision-fatigue · tags
proliferate · search degrades.

**No system:** chaos · 30+ minutes/week lost to "where did I put
that?"

PARA: 4 folders. 5-second filing. Action-oriented. Compounds.

---

## The 1-tool rule

Don't try to run PARA across 12 tools. Pick ONE primary tool
(usually Notion). Run PARA there. Other tools sync into it.

For Purely Personal:
- **Notion** is the PARA hub (Projects + Areas + Archive)
- **Drive** mirrors Resources
- **Gmail labels** mirror Areas (5 engines)
- **Local Documents** is for active drafts only

Single source of truth · multiple windows into it.
